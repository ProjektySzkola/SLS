Kompletna rozpiska plików JS, funkcji, zależności cross-file, tabel Supabase i stanów globalnych. Vanilla JS + Supabase PostgreSQL. Brak frameworków.

Warstwa API — api(path)
ℹ️

api(path) sprawdza ENDPOINT_MAP (4 predefiniowane ścieżki). Dla pozostałych ścieżek parsuje URL i buduje zapytania Supabase dynamicznie przez matchEndpoint(). Zwraca dane lub null przy błędzie (błąd logowany do konsoli).

🐛 Znane problemy i poprawki
✅

NAPRAWIONE — normFmt() duplikat: Było zdefiniowane w 4 plikach (globals, turniej, sport, planowanie + guard w rozstawieniu). Przeniesione do admin-globals.js jako jedyne źródło prawdy z eksportem window.normFmt. Pozostałe pliki mają komentarz zastępczy.

✅

NAPRAWIONE — svEndLeague() raw insert: supabase.from('matches').insert() zastąpiony createMatch() z globals. Zysk: loader, spójność architektury, poprawna obsługa team_id vs id w obiektach z tabeli standings.

✅

NAPRAWIONE — srSave() raw query: DELETE + INSERT na tabeli seeding zastąpione wywołaniem saveSeeding(). wireRankingSettingsEvents() — raw upsert zastąpiony saveTournamentSettings().

✅

NAPRAWIONE — showAddPlayerModal() boolean bug: Pola is_captain, rodo_consent, participation_consent były wstawiane jako 1/0 (integer) zamiast true/false (PostgreSQL boolean). Przepisane na createPlayer().

✅

NAPRAWIONE — string vs number mismatch w matchExists() / _matchKey() / normSeed(): Supabase zwracał team_id i id z bazy jako stringi w niektórych kontekstach (np. po join z tabelą seeding), podczas gdy lokalne obiekty drużyn używały typów numerycznych. Skutki: (1) _matchKey() budował klucze np. "Piłka Nożna|liga|NaN|5" gdy id było undefined, (2) matchExists() nie wykrywało istniejących meczów przy porównaniu string "3" vs number 3, (3) drużyna pojawiała się jednocześnie w slocie rozstawienia i w puli. Poprawka: parseInt(..., 10) dodane w normSeed(), _matchKey(), matchExists() (admin-planowanie.js) oraz w srInitWorkspace() (admin-rozstawienie.js).

✅

NAPRAWIONE — brak synchronizacji _serverMatchKeys w openGenWizard(): Przy każdym otwarciu generatora meczy _serverMatchKeys (zbiór kluczy meczów pobranych z bazy) nie był aktualizowany — zawierał tylko dane z momentu inicjalizacji planera. Skutki: po zapisaniu rozstawienia i otwarciu generatora, matchExists() nie wykrywał meczów utworzonych w poprzednich sesjach → generator oferował już istniejące mecze jako „nowe" lub pokazywał niepoprawną liczbę „+X nowych". Poprawka: openGenWizard() pobiera teraz /matches równolegle z formatem i przebudowuje _serverMatchKeys ze świeżych danych bazy przed wyświetleniem podglądu (admin-planowanie.js).

✅

NAPRAWIONE — CSS vs JS mismatch w klasyfikacji strzelców: Poprzedni CSS zawierał klasy .sv-scorers-section, .sv-sc-rank-cell itp., które nie istniały w HTML generowanym przez svBuildScorers(). CSS przepisany z prawdziwych klas: .sv-sc-medal, .sv-sc-name, .sv-sc-pts--main, .sv-sc-bar-wrap itd.
